# DECISIONS.md — 技术决策记录

## 决策 1：包管理器
- **决策**: 使用 npm
- **原因**: pnpm 不可用，npm 11.x 可用
- **日期**: 2026-06-30

## 决策 2：日历实现方案
- **决策**: FullCalendar 用于月视图网格 + 自定义 React 组件用于周/日视图
- **原因**: FullCalendar 月视图日历网格成熟稳定；周视图和日视图需要三时段布局，FullCalendar 难以满足，自定义实现更灵活

## 决策 3：本地演示模式
- **决策**: 使用 IndexedDB (via idb) 作为本地数据存储
- **原因**: 在没有 Supabase 凭证时提供完整本地功能；通过数据访问接口抽象，后续切换 Supabase 无需大改

## 决策 4：数据访问层架构
- **决策**: 定义 Repository 接口，分别提供 Supabase 和 IndexedDB 实现
- **原因**: 支持本地演示模式和云端模式的平滑切换

## 决策 5：项目子目录
- **决策**: 在 `my-plan-calendar/` 子目录中创建项目
- **原因**: 当前工作区已有其他项目，避免冲突

## 决策 6：默认标签初始化责任方 (2026-07-02)
- **决策**: 由前端 Hook `useInitializeDefaultTags()` 作为唯一初始化方
- **原因**:
  - Supabase 数据库触发器（`handle_new_user`）已在 migration 003 中删除
  - `setupNewUser()` 在注册时也会尝试创建，但使用 upsert 语义
  - 前端 Hook 使用按名称检查和持久化标记确保幂等性
  - 避免多层级重复初始化导致的标签重复
- **替代方案**: 仅由数据库触发器负责 — 被否决，因为触发器在 migration 003 中已被移除且应用层初始化更可控

## 决策 7：标签唯一性规则 (2026-07-02)
- **决策**: 按 `(user_id, LOWER(TRIM(name)))` 建立唯一约束
- **原因**:
  - 防止同一用户创建名称重复的标签
  - 大小写不敏感和去除空格避免视觉上的重复
  - 中文名称保持原值不做额外转换
  - 使用 unique index 而非 unique constraint 以便更灵活地控制
- **迁移**: `supabase/migrations/004_tag_deduplication.sql`

## 决策 8：重复标签合并策略 (2026-07-02)
- **决策**: 保留最早创建的标签（按 `created_at` 升序），迁移关联任务后删除重复项
- **原因**:
  - 最早创建的标签通常是首次初始化产生的，更稳定
  - 不覆盖用户自定义的颜色和设置（保留 earliest 的所有属性）
  - 在事务中执行，确保一致性
  - 可重复运行，再次运行不删除正常数据
  - 仅合并名称规范化后相同的标签，不影响不同名称的标签

## 决策 9：用户删除默认标签后不自动恢复 (2026-07-02)
- **决策**: 使用 `user_preferences.tags_initialized` 持久化标记，标记 set 后不再自动创建默认标签
- **原因**:
  - 尊重用户选择，删除的标签不应自动恢复
  - 只有全新账户首次初始化时创建默认标签
  - 在标签管理界面增加「恢复默认标签」按钮（规划中）供用户主动恢复
- **约束**: 该字段不是 `user_preferences` 表的标准列，通过 IndexedDB 存储任意键和 Supabase `upsert` 的宽松模式实现

## 决策 10：分类视图复用标签 (2026-07-03)
- **决策**: 分类视图直接使用现有标签（tags）作为分类依据，不创建独立的"分类"实体
- **原因**:
  - 标签已包含名称、颜色、图标、排序等所有分类所需属性
  - 避免数据重复和不一致
  - 用户在标签管理中的任何修改自动反映到分类视图
  - 减少数据库迁移和 Repository 变更
- **替代方案**: 创建独立的 categories 表 — 被否决，会增加维护成本且用户需管理两套分类体系

## 决策 11：分类视图不复制事项数据 (2026-07-03)
- **决策**: 分类视图是事项数据的另一种呈现方式，不创建新表或数据副本
- **原因**:
  - 保持单一数据源原则
  - 事项的标签变更在时间视图和分类视图间自动同步
  - 减少数据不一致风险

## 决策 12：评价历史分页策略 (2026-07-03)
- **决策**: 前端分页（加载全部 + 前端切片），而非后端游标分页
- **原因**:
  - 每日评价数据量小（每人每天最多一条）
  - 即使一年也仅 365 条记录
  - 简化 Repository 接口，不需要额外的分页参数
  - 前端 filter + slice 足够应对这个规模
- **替代方案**: 后端分页 — 当前数据规模下过度设计，未来数据量大时可扩展

## 决策 13：presentationMode 与 calendarView 状态分离 (2026-07-03)
- **决策**: 使用两个独立的状态变量 `presentationMode: 'time' | 'category'` 和 `calendarView: 'month' | 'week' | 'day'`
- **原因**:
  - 语义清晰：presentationMode 表示"如何组织"，calendarView 表示"时间粒度"
  - 切换到"按分类"时不丢失用户上次使用的时间视图（月/周/日）
  - 切换回"按时间"时自动恢复之前的时间视图
  - 避免在 CalendarView 类型中混入非时间视图的值
- **持久化**: localStorage key `plan-calendar-presentation-mode`

## 决策 14：手机端使用内联"移动到"菜单 (2026-07-03)
- **决策**: 在事项卡片上直接提供"移动到"下拉菜单，而非依赖长按拖拽
- **原因**:
  - 移动端拖拽体验不稳定，尤其跨分类拖拽
  - 下拉菜单提供明确的分类选择，避免误操作
  - 桌面端同样可用，提供一致的交互方式
  - 降低 @dnd-kit 在移动端的适配复杂度

## 决策 15：UI 动效和视觉克制原则 (2026-07-03)
- **决策**: 所有动画 ≤ 200ms，支持 prefers-reduced-motion，不使用大型装饰动画
- **原因**:
  - 这是一个生产力工具，长期使用的安静感比视觉冲击更重要
  - 过渡动画仅用于提供状态变化反馈（保存、删除、切换）
  - 避免分散用户注意力
  - 符合 WCAG 无障碍要求
